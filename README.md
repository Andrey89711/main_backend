# Backend TSZH System

Backend реализует REST API для веб-системы ТСЖ: регистрация и вход пользователей, роли, профили, адреса жильцов, подтверждение адресов, создание и обработка заявок.

## Стек

- Python
- FastAPI
- SQLAlchemy
- MySQL
- PyMySQL
- JWT
- Passlib / bcrypt
- python-dotenv

## Запуск

1. Установить зависимости:

```powershell
cd D:\Python_project\dimas\main_backend
..\ .venv\Scripts\python -m pip install -r requirements.txt
```

2. Создать базу MySQL, например:

```sql
CREATE DATABASE IF NOT EXISTS tszh_db
CHARACTER SET utf8mb4
COLLATE utf8mb4_unicode_ci;
```

3. Заполнить `.env`:

```env
DB_USER=root
DB_PASSWORD=root
DB_HOST=localhost
DB_PORT=3306
DB_NAME=tszh_db

DATABASE_URL=mysql+pymysql://${DB_USER}:${DB_PASSWORD}@${DB_HOST}:${DB_PORT}/${DB_NAME}
```

4. Запустить API:

```powershell
python -m uvicorn app.main:app --reload
```

API будет доступен по адресу:

```text
http://127.0.0.1:8000
```

Swagger:

```text
http://127.0.0.1:8000/docs
```

## База данных

Проект пока не использует Alembic. Таблицы создаются при старте приложения через:

```python
Base.metadata.create_all(bind=engine)
```

Также при старте выполняются служебные действия:

- добавляется колонка `personal_account` в `addresses`, если ее еще нет;
- создаются базовые роли;
- старые адреса из `users.address_id` переносятся в `user_addresses`.

## Основные таблицы

### users

Хранит учетные записи пользователей:

- `id`
- `full_name`
- `email`
- `phone`
- `password_hash`
- `role`
- `is_active`
- `address_id` — legacy-поле, новая логика адресов работает через `user_addresses`

### roles

Хранит роли пользователей:

- `resident` — жилец
- `dispatcher` — диспетчер
- `admin` — администратор
- `executor` — исполнитель

### addresses

Хранит адрес объекта недвижимости:

- `id`
- `street`
- `house`
- `apartment`
- `personal_account`

### user_addresses

Связующая таблица между пользователями и адресами:

- `id`
- `user_id`
- `address_id`
- `is_primary`
- `is_verified`
- `created_at`

Один пользователь может иметь несколько адресов. Один адрес может быть связан с несколькими пользователями через отдельные записи связи.

### tickets

Хранит заявки:

- `id`
- `description`
- `status`
- `priority`
- `created_at`
- `resident_id`
- `category_id`
- `address_id`

### comments

Хранит комментарии к заявкам:

- `id`
- `text`
- `ticket_id`
- `user_id`
- `created_at`

## Ролевая логика

### Жилец (`resident`)

Может:

- регистрироваться;
- входить в систему;
- смотреть свои заявки;
- создавать заявки по подтвержденным адресам;
- смотреть и редактировать профиль;
- добавлять, редактировать, удалять свои адреса;
- выбирать основной адрес;
- писать комментарии к заявкам.

### Исполнитель (`executor`)

Пока использует такой же интерфейс, как жилец:

- главная;
- создание заявки;
- профиль;
- просмотр своих заявок.

### Диспетчер (`dispatcher`)

Может:

- видеть все заявки;
- менять статус заявок;
- подтверждать адреса жильцов;
- смотреть свои заявки на главной странице.

Не может:

- создавать заявки.

### Администратор (`admin`)

Может:

- видеть список пользователей;
- назначать роли другим пользователям;
- подтверждать адреса жильцов.

Не может:

- менять свою собственную роль;
- создавать заявки;
- пользоваться страницами заявок, профиля и диспетчера в текущем интерфейсе.

## Авторизация

Авторизация реализована через JWT.

После входа пользователь получает `access_token`. В токене хранится:

- `sub` — id пользователя;
- `email`;
- `role`;
- `exp`.

Для совместимости backend также умеет читать старые токены, где `sub` был email.

## Адресная логика

Адрес можно добавить:

- при регистрации;
- в личном кабинете.

При добавлении адреса указываются:

- улица;
- дом;
- квартира;
- лицевой счет.

После добавления адрес получает статус:

```text
Не подтвержден
```

Создание заявок по неподтвержденному адресу запрещено.

Подтвердить адрес может:

- диспетчер;
- администратор.

Если жилец редактирует адрес, подтверждение сбрасывается, и адрес снова должен быть подтвержден.

Удаление адреса запрещено, если по нему есть активные заявки со статусом не `completed`.

## Логика создания заявки

При создании заявки frontend передает:

- `description`
- `category_id`
- `address_id`

Backend проверяет:

- текущий пользователь не `admin` и не `dispatcher`;
- адрес связан с текущим пользователем;
- адрес подтвержден;
- категория существует или создается дефолтная категория `Общие заявки`.

Если адрес чужой или неподтвержденный, заявка не создается.

## Статусы заявок

Используются статусы:

- `new`
- `in_progress`
- `completed`

Диспетчер может переводить заявку:

- в работу;
- в завершенные.

## API

### Auth

`POST /auth/register`

Регистрирует жильца и создает первый адрес. Адрес сразу связывается с пользователем, но остается неподтвержденным.

`POST /auth/login`

Авторизует пользователя и возвращает JWT.

### Users

`GET /users/`

Список пользователей. Доступно только администратору.

`GET /users/roles`

Список ролей. Доступно только администратору.

`PATCH /users/{user_id}/role`

Изменение роли пользователя. Доступно только администратору.

`GET /users/me`

Профиль текущего пользователя.

`PUT /users/me`

Обновление контактных данных и пароля текущего пользователя.

### Addresses

`GET /addresses/my`

Все адреса текущего пользователя.

`GET /addresses/my/verified`

Только подтвержденные адреса текущего пользователя.

`POST /addresses/my`

Добавление адреса пользователем.

`PUT /addresses/my/{link_id}`

Редактирование собственного адреса. После редактирования адрес становится неподтвержденным.

`PATCH /addresses/my/{link_id}/primary`

Выбор основного адреса.

`DELETE /addresses/my/{link_id}`

Удаление адреса, если по нему нет активных заявок.

`GET /addresses/pending`

Список неподтвержденных адресов. Доступно диспетчеру и администратору.

`PATCH /addresses/{link_id}/verify`

Подтверждение адреса. Доступно диспетчеру и администратору.

### Tickets

`POST /tickets/`

Создание заявки по подтвержденному адресу.

`GET /tickets/`

Список заявок текущего пользователя.

`GET /tickets/all`

Все заявки. Доступно только диспетчеру.

`PATCH /tickets/{ticket_id}/status`

Изменение статуса заявки. Доступно только диспетчеру.

`GET /tickets/{ticket_id}/comments`

Комментарии заявки.

`POST /tickets/{ticket_id}/comments`

Добавление комментария к заявке.

## Ограничения текущей версии

- Alembic не настроен.
- Проверка лицевого счета пока ручная: адрес подтверждается диспетчером или администратором.
- Механизм связанных обращений пока не реализован.
- Журналирование операций добавления и изменения адресов пока не реализовано.
- `users.address_id` оставлено как legacy-поле для совместимости.
